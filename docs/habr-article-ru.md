# AI Toolbox: фреймворк, который превращает ИИ-агентов из хаотичных стажёров в дисциплинированных инженеров

Привет, Хабр.

Терминальные ИИ-агенты — Claude Code, Qwen Code, Cursor, Gemini CLI — за последний год стали по-настоящему мощными. Они читают код, запускают тесты, пишут фичи. Но если дать агенту свободу без ограничений, через 20 минут работы вы получите:

- 4000 строк сырого вывода `npm test` в контексте (и счёт за токены)
- «Давайте перепишем на Rust» — хотя вы договорились оставаться на Go
- Код без тестов, потому что «так быстрее»
- После перезапуска сессии — полная амнезия: «Какую архитектуру мы выбрали?»

Каждый раз одно и то же: вместо разработки вы объясняете агенту заново контекст, правила, решения.

**AI Toolbox** — это open-source фреймворк, который решает эти три проблемы системно: контролирует вывод, сохраняет память между сессиями и принуждает агента следовать инженерным практикам. Один `git clone` — и он работает с любым из 8 ИИ-клиентов.

## Три проблемы, которые мы решаем

### Проблема 1: Context Bloat — раздувание контекста

Агент запускает `pytest` — получает 5000 строк. Читает их все. Платит за каждый токен. Путается в стектрейсах. Теряет фокус на задаче.

**Решение:** `rtk` (Rust Token Killer) — консольный прокси, который перехватывает вывод тяжёлых команд и сжимает его на 60-90%. Лог на 5000 строк превращается в 500-800 строк с сохранением всех ошибок и контекста.

```bash
# Вместо:
pytest                    # 5000 строк, все в контекст

# Агент делает:
rtk pytest                # 500-800 строк, только релевантное
```

### Проблема 2: Project Amnesia — потеря контекста между сессиями

Вы потратили час, объясняя агенту архитектуру. Договорились использовать определённый паттерн. Закрыли терминал. Открыли заново — агент ничего не помнит.

**Решение:** Папка `.agent/memory/` — обычные Markdown-файлы в Git. Не RAG, не векторная база — простые файлы, которые агент читает при каждом старте:

```
.agent/memory/
├── architecture-decisions.md   # ADR: какие решения приняты и почему
├── integration-contracts.md    # API-контракты, схемы данных
├── session-handover.md         # Где остановились, что дальше
├── current-task.md             # Активная задача (из Beads)
├── runbook.md                  # Операционные процедуры
└── active-session.md           # Статус текущей сессии
```

Пример `session-handover.md` после реальной рабочей сессии:

```markdown
## Completed
- REST API endpoint для /users: schema, handler, тесты
- Миграция БД добавлена (migration_003_users.sql)

## In Progress
- Интеграция с auth middleware (50%)

## Next recommended step
- Дописать middleware, тест на 401 Unauthorized
- Запустить полный тест-сюит перед мержем

## Stats
- Tokens saved (rtk): ~12000
- Sub-agents used: 2 (параллельный аудит schema + handler)
```

Следующая сессия начинается не с нуля, а **ровно там, где закончилась предыдущая**.

### Проблема 3: Execution Drift — хаотичная работа без процесса

Агент получает задачу «добавить фичу X» и сразу начинает писать код. Без планирования. Без тестов. Без ревью. Если что-то сломалось — «давайте перепишем всё».

**Решение:** 9-шаговый воркфлоу, закодированный в `.agent/workflows/unified-workflow.md`:

```
TASK → BRAINSTORM → PLAN → ISOLATE → IMPLEMENT → REVIEW → VERIFY → FINISH → CLOSE
```

Каждый шаг — это конкретный чеклист с критериями перехода. Агент не перескакивает шаги: нельзя начать IMPLEMENT без PLAN, нельзя вызвать FINISH без VERIFY.

### Слой 1: Правила (.agent/rules/)

Жёсткие ограничения, которые агент обязан выполнять:

| Правило | Что делает |
|---------|-----------|
| `safety-rules.md` | Запрещает удаление файлов, force-push, перезапись конфигов без подтверждения |
| `tdd-rules.md` | Обязательный цикл RED-GREEN-REFACTOR для любых изменений кода |
| `stack-rules.md` | Запрещает добавлять зависимости вне согласованного стека |
| `mcp-rules.md` | Контролирует доступ к внешним ресурсам через MCP |
| `testing-rules.md` | Запрещает объявлять задачу выполненной без верификации |
| `status-reporting.md` | Обязывает отчитываться на каждом шаге воркфлоу |

### Слой 2: Воркфлоу (.agent/workflows/)

Стандартизированные процедуры для типовых задач:

| Воркфлоу | Шаги | Когда используется |
|----------|------|--------------------|
| `unified-workflow.md` | 9 шагов: TASK → CLOSE | Новые фичи, создание компонентов |
| `bug-fix.md` | 5 фаз: Repro → Identify → Fix → Verify → Record | Баг-фиксы |
| `code-review.md` | Чеклист из 5+ пунктов | Перед мержем |
| `multi-agent.md` | Spawn → Execute → Synthesize | 3+ независимых подзадачи |
| `branch-finish.md` | Merge/PR решение + cleanup | Завершение ветки |

### Слой 3: Хуки и скрипты (.agent/scripts/)

Автоматические перехватчики, которые работают без вашего участия:

**Pre-command хук** — перехватывает тяжёлые команды до выполнения:

```bash
# Агент пишет: pytest
# Хук перехватывает:
# 🚨 AI Toolbox Heavy Command Detected!
# Please use 'rtk' wrapper: rtk pytest
```

Список перехватываемых команд: `python`, `pytest`, `cargo`, `go`, `npm run`, `mvn`, `gradle`, `docker`, `java`, `ansible-playbook` и другие. Также перехватывает `cat`/`tail`/`head` на `.log`-файлах.

**Verify-commit хук** — блокирует коммиты кода без тестов:

```bash
# Агент коммитит .py файл без test_*.py
# 🚨 AI Toolbox TDD Enforcement: Code changes without test updates.
#    Staged code files:
#      src/handler.py
#    To fix: Stage corresponding test files and commit again.
#    To skip (emergency only): Include 'tdd-skip' in commit message.
```

**Hook-stop** — при завершении сессии автоматически:
- Запускает `sync-task` для синхронизации задач
- Записывает прогресс в `session-handover.md`
- Показывает статистику: сколько раз использовались rtk, Beads, MCP

## 3-Tier система: один фреймворк, 8 клиентов

Ключевая архитектурная идея: ИИ-клиенты имеют **разные возможности**. Claude Code поддерживает хуки и мульти-агентов. Gemini CLI — нет. Toolbox адаптируется через трёхуровневую систему:

| Клиент | Tier | Хуки | Мульти-агент | Правила | Router-файл |
|--------|------|------|-------------|---------|-------------|
| Claude Code | Full | да | Agent Teams | да | `CLAUDE.md` |
| Antigravity | Full | да | Agent Manager | да | `SKILL.md` |
| Qwen Code | Full | да | SubAgents | да | `QWEN.md` |
| Cursor | Standard | да | Background | да | `.cursorrules` |
| RooCode / Cline | Standard | да (v3.36+) | нет | да | `.clinerules` |
| Windsurf | Standard | да | нет | да | `.windsurfrules` |
| Gemini CLI | Basic | нет | нет | да | `GEMINI.md` |
| Aider | Basic | нет | нет | да | `CONVENTIONS.md` |

**Как это работает:**

- **Full-tier** (Claude Code, Qwen Code, Antigravity): Хуки принудительно блокируют опасные действия. Мульти-агентная оркестрация доступна. Полная автоматизация через sync/handover скрипты.
- **Standard-tier** (Cursor, Cline, Windsurf): Хуки работают для sync/handover. Мульти-агентов нет. Правила читаются из файлов.
- **Basic-tier** (Gemini CLI, Aider): Хуков нет — правила работают как **мягкие рекомендации**. Память и правила доступны, но не принудительны.

Каждый клиент читает свой router-файл, который указывает на `AGENT.md` и сообщает агенту его tier. Один `.agent/` — одни правила для всех.

## Boot Sequence: как агент «просыпается»

Каждая сессия начинается с 6-шаговой процедуры загрузки (AGENT.md, секция 2):

```
1. Environmental Check  → .agent/ существует? rtk установлен? Beads установлен?
2. Context Recovery     → Читает architecture-decisions.md + integration-contracts.md
3. WIP Check            → Читает session-handover.md — что не закончено
4. Task Sync            → Запускает sync-task.sh → обновляет current-task.md
5. Session Init         → Инициализирует active-session.md
6. Summarization        → Краткий отчёт: контекст, состояние, доступные инструменты
```

Вот что агент выводит после загрузки:

```
✅ AI Toolbox Active
  → rtk: installed
  → Beads: installed
  → MCP: configured (context7, sequential-thinking)

📋 Context recovered:
  - ADR-0001: REST API на Go + PostgreSQL (accepted)
  - ADR-0002: JWT auth вместо сессий (accepted)

📋 WIP: Интеграция auth middleware (50%)
   Next step: тест на 401 Unauthorized

📋 Current task: BD-042 "Auth middleware integration" (in-progress)

Ready to continue.
```

10-30 секунд вместо 2-3 минут диалога «помнишь, мы договорились...».

## Ансамбль инструментов: как всё работает вместе

Toolbox — это не один инструмент, а координированный ансамбль из пяти:

| Инструмент | Решает | Установка |
|------------|--------|-----------|
| [rtk](https://github.com/rtk-ai/rtk) | Сжатие вывода (60-90% экономии токенов) | `cargo install --git https://github.com/rtk-ai/rtk` + `rtk init -g` |
| [Beads](https://github.com/steveyegge/beads) | Трекинг задач вне контекста чата | `go install github.com/steveyegge/beads/cmd/bd@v0.63.3` + `bd init` |
| [Superpowers](https://github.com/obra/superpowers) | Инженерная дисциплина (TDD, планирование, ревью) | Встроено в `.agent/rules/` — установка не нужна |
| [Template Bridge](https://github.com/maslennikov-ig/template-bridge) | 413+ экспертных шаблонов для специализированных задач | Claude Code plugin или `npx` |
| [MCP-серверы](https://modelcontextprotocol.io/) | Внешние ресурсы: документация, GitHub, веб | `claude mcp add ...` |

### Как они координируются

Рассмотрим полный цикл на примере «добавить REST-эндпоинт»:

```
Пользователь: "Добавь REST API endpoint /users"
  │
  ├─ 1. TASK        → Beads: bd create "REST endpoint /users" → задача в графе
  ├─ 2. BRAINSTORM  → Superpowers: анализ, 2 подхода (REST vs GraphQL) → решение: REST
  │                  → ADR записан в architecture-decisions.md
  ├─ 3. PLAN        → 3 подзадачи: schema, handler, тесты
  │
  ├─ 4-5. IMPLEMENT (для каждой подзадачи):
  │   ├─ RED:         Пишем тест → rtk test → FAILS ✓
  │   ├─ GREEN:       Пишем минимальный код → rtk test → PASSES ✓
  │   │               (rtk сжимает вывод тестов — экономия ~60% токенов)
  │   ├─ REFACTOR:    Чистим код, тесты остаются зелёными
  │   └─ COMMIT:      verify-commit проверяет: есть ли тест-файлы → OK
  │
  ├─ 6. REVIEW    → code-review.md чеклист: тесты, lint, debug statements
  ├─ 7. VERIFY    → rtk test + rtk lint → clean
  ├─ 8. FINISH    → session-handover.md обновлён
  └─ 9. CLOSE     → Beads: bd close → следующая задача готова
```

На каждом шаге работает свой инструмент. Нет ручной оркестрации — агент следует воркфлоу автоматически.

## TDD Enforcement: не рекомендация, а требование

Большинство фреймворков «рекомендуют» TDD. Toolbox **принуждает**:

### Цикл RED-GREEN-REFACTOR

```
1. RED       — Пишем тест. Описываем ожидаемое поведение.
2. VERIFY RED — Запускаем: rtk test → тест ДОЛЖЕН упасть.
               Если тест прошёл — тест неправильный. Переписать.
3. GREEN     — Пишем минимальный код, чтобы тест прошёл.
               Не больше. Никаких «заодно добавлю...»
4. VERIFY GREEN — rtk test → тест ДОЛЖЕН пройти.
                  Все существующие тесты тоже ДОЛЖНЫ пройти.
5. REFACTOR  — Чистим код. Тесты остаются зелёными.
6. COMMIT    — verify-commit хук проверяет: если в коммите есть .py/.ts/.go,
               должны быть и тест-файлы. Иначе коммит заблокирован.
```

### Анти-паттерны, которые Toolbox предотвращает

| Анти-паттерн | Почему плохо | Что делает Toolbox |
|-------------|-------------|-------------------|
| Код без теста | Нет страховки от регрессий | `verify-commit` блокирует коммит |
| Пропуск «verify red» | Тест может быть бессмысленным | tdd-rules.md: шаг обязателен |
| Больше кода, чем нужно | Over-engineering, мёртвый код | Правило: минимальный код для прохождения теста |
| Рефакторинг до зелёных тестов | Невозможно понять, что сломано | Правило: сначала GREEN, потом REFACTOR |
| Мок всего подряд | Тесты не отражают реальность | Правило: интеграционные тесты предпочтительнее |

Аварийный обход: `tdd-skip` в сообщении коммита — для экстренных хотфиксов. Но это задокументированное исключение, а не норма.

## Мульти-агентная оркестрация

Full-tier клиенты (Claude Code, Qwen Code, Antigravity) поддерживают параллельное выполнение задач через суб-агентов.

### Когда активируется

Контекстно-зависимая активация (AGENT.md, секция 13): когда `bd ready` показывает 3+ независимых задач, или когда агент сам определяет, что задача декомпозируется на независимые части.

### Паттерн оркестрации

```
Оркестратор получает задачу → разбивает на 2-5 независимых подзадач

🤖 Spawning Agent 1/3: Аудит bootstrap.sh (general-purpose)
🤖 Spawning Agent 2/3: Аудит bootstrap.ps1 (general-purpose)
🤖 Spawning Agent 3/3: Проверка ссылок (Explore)

    [агенты работают параллельно]

🤖 Agent 1/3 complete: 5 issues found
🤖 Agent 2/3 complete: 3 issues found
🤖 Agent 3/3 complete: all links valid

Оркестратор → синтез результатов → единый отчёт
```

### Типы агентов

| Тип | Для чего | Стоимость |
|----|---------|-----------|
| `general-purpose` | Глубокий анализ, сложные исследования | Высокая (полный контекст) |
| `Explore` | Быстрый поиск файлов, ключевых слов | Низкая |

### Правила координации

- Подзадачи не должны зависеть друг от друга (иначе — последовательно)
- Промпт каждого суб-агента < 500 токенов
- Результаты записываются в `.agent/memory/active-session.md`
- При ошибке: одна повторная попытка, затем fallback на последовательное выполнение

## MCP-интеграция: расширяем возможности агента

Model Context Protocol (MCP) — открытый стандарт для подключения внешних ресурсов к ИИ-клиентам. Toolbox поддерживает MCP через профильную систему:

### Профили

| Профиль | Серверы | Когда использовать |
|---------|---------|-------------------|
| **Minimal** | context7, sequential-thinking | Быстрые задачи, ограниченный бюджет |
| **Developer** | + filesystem, fetch | Ежедневная работа (рекомендуется) |
| **Full** | + github, memory | Полная проектная работа (нужен GITHUB_TOKEN) |

### Установка (пример для Claude Code)

```bash
# Minimal — документация + рассуждения
claude mcp add context7 npx -y @upstash/context7-mcp
claude mcp add sequential-thinking npx -y @modelcontextprotocol/server-sequential-thinking

# Developer — + доступ к файлам + web-контент
claude mcp add filesystem npx -y @modelcontextprotocol/server-filesystem .
claude mcp add fetch npx -y @modelcontextprotocol/server-fetch
```

Конфигурации для всех 8 клиентов — в `.agent/templates/mcp/`. Правила безопасности: `mcp-rules.md` — read-only по умолчанию, никаких захардкоженных токенов, только проверенные MCP-серверы.

## Контекстно-зависимая активация навыков

Агент не просто следует правилам — он **автоматически активирует** нужный навык в зависимости от контекста задачи (AGENT.md, секция 13):

| Триггер | Активируемый навык | Что происходит |
|---------|-------------------|---------------|
| Команда `rtk test`, `pytest` и т.д. | TDD Rules | Принудительный цикл RED-GREEN-REFACTOR |
| Задача содержит «fix», «bug», «issue» | Bug-Fix Workflow | 5 фаз: Repro → Identify → Fix → Verify → Record |
| Задача содержит «refactor», «migrate» | Code Review | Чеклист перед завершением |
| Задача содержит «feature», «build» | Unified Workflow | Полный 9-шаговый процесс |
| 3+ независимых подзадач | Multi-Agent | Параллельные суб-агенты |
| Незнакомая технология | Template Bridge | Экспертный шаблон из 413+ вариантов |
| Нужны внешние данные | MCP Rules | Запрос через настроенные MCP-серверы |

Агент анонсирует активацию:

```
📋 Skill activated: TDD Rules — RED phase
📋 Workflow: Bug-Fix — Phase 1/5: REPRODUCE
💡 Template available: devops-infrastructure/kubernetes
```

## Safety Rules: что агент не имеет права делать

Не только процесс, но и границы. `.agent/rules/safety-rules.md` определяет:

**Запрещено без явного подтверждения пользователя:**
- Удаление файлов или директорий
- Перезапись большой части репозитория
- Замена ключевых технологий
- Force-push или перезапись Git-истории
- Удаление тестов, валидации или safety-проверок

**Зоны повышенной осторожности:**
- Миграции БД
- Аутентификация / авторизация
- Секреты и credentials
- Deployment-конфигурация
- Деструктивные shell-команды

**Правило предположений:** Если агент что-то выводит, но не проверил — он обязан это указать. Необратимые действия на основе предположений запрещены.

## CI/CD: автоматические проверки

Toolbox включает GitHub Actions workflow (`.github/workflows/ci.yml`), который проверяет:

1. **Tier-бейджи** — каждый router-файл должен содержать `-- Tier: X`
2. **ADR-формат** — architecture-decisions.md должен содержать `### ADR-XXXX` записи
3. **Битые ссылки** — все внутренние Markdown-ссылки должны указывать на существующие файлы
4. **JSON-валидация** — client-capabilities.json должен быть валидным JSON
5. **Паритет скриптов** — bootstrap.sh и bootstrap.ps1 должны создавать одинаковую структуру

Плюс шаблоны issue (Bug Report, Feature Request) для стандартизации обратной связи.

## Чего Toolbox не умеет (и не пытается)

Честность важнее маркетинга:

- **Не заменяет человеческое ревью.** `code-review.md` ловит очевидные баги и стилистические проблемы. Архитектурные решения и edge cases — за человеком.
- **TDD enforced, но не абсолютен.** `tdd-skip` в сообщении коммита обходит проверку. Это задокументированный аварийный выход, а не дыра в системе.
- **Basic-tier клиенты работают без хуков.** Gemini CLI и Aider получают правила как «мягкие рекомендации» — агент может их проигнорировать. Память и воркфлоу доступны, но не принудительны.
- **Не экономит токены на генерации.** rtk сжимает *вывод команд*, а не стоимость генерации кода моделью.
- **GUI-клиенты (Cursor, Cline, Windsurf) требуют проверки хуков.** `setup.sh/ps1` генерирует конфигурацию автоматически, но фактическое выполнение зависит от реализации каждого клиента.

## Установка

### Способ 1: Через ИИ-агента (рекомендуемый)

Скопируйте этот промпт в вашего терминального агента:

```text
Follow the setup instructions here to initialize the AI Toolbox environment:
https://raw.githubusercontent.com/mrAibo/AI_Toolbox/main/INSTALL.md
```

Агент сам скачает `.agent/`, создаст `AGENT.md`, запустит `bootstrap.sh/ps1`, настроит хуки, определит стек проекта и инициализирует память.

### Способ 2: setup-скрипт

```bash
# Linux / macOS / Git Bash
bash .agent/scripts/setup.sh

# Windows PowerShell
powershell -ExecutionPolicy Bypass -File .agent/scripts/setup.ps1
```

Скрипт сделает следующее:
1. **Найдёт всех установленных ИИ-клиентов** — Claude Code, Qwen Code, Gemini CLI, Aider, Cursor, Cline, Windsurf
2. **Запустит bootstrap** — создаст `.agent/memory/`, `.agent/rules/`, router-файлы
3. **Определит стек проекта** — Node.js, Rust, Python, Go, Java
4. **Предложит установить rtk** — оптимизатор вывода (`cargo install --git https://github.com/rtk-ai/rtk`, ⚠️ не `cargo install rtk` — это другой проект)
5. **Предложит установить Beads** — трекер задач (`go install`)
6. **Настроит MCP-серверы** — по выбранному профилю
7. **Зарегистрирует хуки** — для каждого обнаруженного клиента

### Способ 3: Клонирование и ручная настройка

```bash
git clone https://github.com/mrAibo/AI_Toolbox.git
cd AI_Toolbox
bash .agent/scripts/bootstrap.sh
```
