# AI Toolbox: Как мы заставили ИИ-агентов работать по нашим правилам, а не наоборот

Привет, Хабр.

Скажу честно: когда я первый раз запустил Claude Code в своём проекте, он работал примерно как стажёр в первый день. Умный, энтузиаст, но:

— Забыл проверить существующие зависимости и начал писать код, который конфликтовал с тем, что уже было
— Запустил `npm test`, получил 4000 строк вывода, прочитал их все (заплатив за каждый токен) и запутался
— Через три сессии забыл, какую архитектуру мы с ним договорились, и начал придумывать файлы, которых нет
— Писал код без тестов, потому что «так быстрее», хотя я просил TDD

И каждый раз я тратил время не на разработку, а на то, чтобы объяснять агенту заново: «Помнишь, мы договорились X?», «Проверь сначала зависимости», «Где тесты?»

Надоело. Поэтому я собрал **AI Toolbox** — набор правил, хуков и скриптов, которые делают то, что хороший техлид делает для человека-разработчика: **не дают накосячить до того, как это случилось**.

---

## Что вообще делает Toolbox?

Представьте, что ваш ИИ-агент приходит в новый проект. Toolbox делает три вещи:

### 1. Даёт ему память

Агент читает папку `.agent/memory/` и понимает:

- **Какие решения уже приняты** — ADR-файлы с архитектурными решениями. Агент не предложит «а давайте перепишем на Rust» если мы уже задокументировали, что остаёмся на Go.
- **Что сейчас в работе** — `current-task.md` синхронизируется с Beads (CLI-трекером задач от Стива Йегги). Никаких «что мы делаем?» — агент видит список задач.
- **Где остановились** — `session-handover.md` записывает прогресс, блокеры, следующие шаги. Новая сессия начинается не с нуля.

Это не RAG. Это обычные Markdown-файлы в Git. 100% гарантия что агент прочитает именно то, что нужно.

### 2. Даёт ему правила

Папка `.agent/rules/` — это как coding guidelines, только ИИ читает их **обязательно**, а не «когда вспомнит»:

- **safety-rules.md** — какие команды опасны, что нельзя делать без подтверждения
- **tdd-rules.md** — RED-GREEN-REFACTOR, обязательно. Хук `verify-commit` **блокирует** коммит кода без тестов (с аварийным обходом `tdd-skip` для срочных фиксов)
- **stack-rules.md** — какие зависимости допустимы, какие нет
- **mcp-rules.md** — как работать с MCP-серверами, когда можно вызывать внешние ресурсы
- **status-reporting.md** — когда и как агент отчитывается о прогрессе

Плюс одиннадцать workflow-файлов: bug-fix, code-review, multi-agent, unified-workflow... Агент не изобретает процесс — он следует задокументированному.

### 3. Не даёт нарушить правила

Это хуки — Shell и PowerShell скрипты, которые перехватывают команды до выполнения.

Агент хочет запустить `npm test` напрямую? Хук видит тяжёлую команду и пишет:

> ⚠️ Используй `rtk npm test` — это сэкономит токены.

Хочет закоммитить код без тестов? `verify-commit` блокирует:

> 🚨 Code changes without test updates. Stage corresponding test files and commit again.

Завершает сессию? `hook-stop` автоматически:
- Синхронизирует состояние задач через `sync-task`
- Записывает прогресс в `session-handover.md`
- Показывает статистику использования инструментов за сессию

Всё это без вашего участия. Агент просто работает — а система следит чтобы по правилам.

## Что под капотом

### rtk — оптимизатор вывода

[rtk](https://github.com/rtk-ai/rtk) на Rust перехватывает вывод тяжёлых команд и вычищает мусор. Лог на 5000 строк сжимается до 500–800. Это не главная фича проекта, а вспомогательный инструмент — агент не тонет в логах и быстрее находит ошибку.

### Beads — менеджер задач

[Beads](https://github.com/steveyegge/beads) от Стива Йегги хранит дерево задач прямо в Git. `sync-task` скрипт выгружает состояние в `current-task.md` при каждом старте сессии и после каждой команды. Агент всегда знает контекст.

### Template Bridge — 413+ экспертных шаблонов

Когда встроенных навыков (TDD, планирование, дебаг) недостаточно, [Template Bridge](https://github.com/maslennikov-ig/template-bridge) даёт доступ к специализированным шаблонам: Kubernetes, GraphQL, async Rust, что угодно. Агент не гадает как работать с незнакомой технологией — берёт готовый экспертный шаблон.

### Superpowers — инженерная дисциплина

Наработки [obra / Superpowers](https://github.com/obra/superpowers) дают процесс: сначала план, потом код. Сначала тест, потом реализация. Agent следует 9-шаговому воркфлоу (TASK → BRAINSTORM → PLAN → ISOLATE → IMPLEMENT → REVIEW → VERIFY → FINISH → CLOSE) а не пишет код хаотично.

### MCP-серверы

Toolbox настраивает MCP-серверы (context7 для документации, sequential-thinking для сложных рассуждений) через `.agent/config/client-capabilities.json`. Правила работы с MCP описаны в `mcp-rules.md`: когда можно вызывать внешние ресурсы, как обрабатывать ошибки, как координировать MCP с мульти-агентными сценариями.

## Автоматизация: что работает без вас

Весь смысл Toolbox в том, что **вы не должны помнить о рутине**. Система делает это сама:

| Что | Как |
|---|---|
| **Восстановление контекста** | Boot Sequence: при старте агент читает ADR, контракты, handover, задачи. 10–30 секунд вместо 2–3 минут диалога |
| **Синхронизация задач** | `sync-task` при каждом старте и завершении сессии. Beads ↔ текстовый файл |
| **Защита от тяжёлых команд** | Pre-command хук видит `python`, `pytest`, `cargo`, `go build`, `npm run`, `mvn`, `gradle`, `docker`, `java` и другие тяжёлые команды без `rtk` → предупреждение. Аналогично для чтения логов: `cat`, `tail`, `head` на `.log` файлах |
| **TDD enforcement** | `verify-commit`: код без тестов → коммит заблокирован. Обход: `tdd-skip` в сообщении коммита |
| **Session handover** | `hook-stop`: при завершении сессии автоматически записывает прогресс |
| **Мульти-агентная оркестрация** | Когда `bd ready` показывает 3+ независимых задач — агент получает рекомендацию использовать `multi-agent.md` с распараллеливанием |
| **Активация навыков** | Агент читает триггеры в `AGENT.md` §13: «fix/bug» → bug-fix workflow, «refactor» → code-review, 3+ задач → multi-agent |

## Один Toolbox — любые агенты

Toolbox не привязан к одному клиенту. Мы тестировали:

- **Claude Code, Qwen Code, Gemini CLI** — полная поддержка через терминальные хуки, `rtk`, мульти-агентную оркестрацию
- **Cursor, Windsurf, Cline** — читают `.cursorrules`, `.windsurfrules`, `.clinerules` как точки входа. Хуки настраиваются автоматически через `setup.ps1` (PowerShell на Windows)
- **Aider** — базовый режим: читает `AGENT.md` и следует правилам

Если агент умеет читать файлы из репозитория и выполнять команды — он совместим с Toolbox.

## Чего Toolbox не умеет

Честность важнее маркетинга:

- **Не заменяет человеческое код-ревью.** Есть `review` skill — проверяет correctness, security, quality. Но ИИ-ревью ловит очевидные баги, а архитектурные ошибки и edge cases может пропустить. Финальное ревью — за человеком.
- **TDD enforced, но не абсолютен.** Хук `verify-commit` блокирует коммиты без тестов. Но есть `tdd-skip` для экстренных случаев. На практике агент следует TDD гораздо чаще когда правила записаны и enforced.
- **GUI-агенты требуют проверки хуков.** `setup.ps1` автоматически создаёт конфигурацию хуков для Cursor, Cline и Windsurf (с PowerShell-путями). Но фактическое выполнение зависит от реализации каждого агента — некоторые могут не читать наши JSON-файлы. В таком случае хуки можно подключить вручную.
- **Не экономит токены за саму модель.** rtk сжимает логи команд, но не стоимость генерации кода.

## Установка

### Способ 1: Через ИИ-агента (рекомендуемый)

Скопируйте этот промпт в вашего терминального ИИ-агента (Claude Code, Qwen Code, Gemini CLI):

```text
Follow the setup instructions here to initialize the AI Toolbox environment:
https://raw.githubusercontent.com/mrAibo/AI_Toolbox/main/install.md
```

Агент сам скачает `.agent/`, создаст `AGENT.md`, настроит хуки и инициализирует память.

### Способ 2: Вручную

Если хотите контролировать процесс — запустите скрипт настройки напрямую:

```bash
# Linux / macOS / Git Bash на Windows
bash .agent/scripts/setup.sh

# Windows PowerShell
powershell -ExecutionPolicy Bypass -File .agent/scripts/setup.ps1
```

`setup.sh` / `setup.ps1` сделают следующее:
1. **Найдут всех установленных ИИ-агентов** — Claude Code, Qwen Code, Gemini CLI, Aider, Cursor, Cline, Windsurf
2. **Запустят bootstrap** — создадут `.agent/memory/`, `.agent/rules/`, файлы маршрутизаторов (`.cursorrules`, `.windsurfrules` и т.д.)
3. **Определят стек проекта** — Node.js, Rust, Python, Go, Java
4. **Предложат установить rtk** — оптимизатор вывода (через `cargo install`)
5. **Предложат установить Beads** — менеджер задач (через `go install`)
6. **Настроят MCP-серверы** — context7, sequential-thinking
7. **Зарегистрируют хуки** для каждого обнаруженного клиента

Оба скрипта кроссплатформенные и работают на Linux, macOS и Windows.
